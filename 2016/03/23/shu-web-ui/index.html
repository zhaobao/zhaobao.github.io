术·从UI设计图到代码尺寸

前端UI

rem values were invented in order to sidestep the compounding problem. rem values are relative to the root html element, not the parent element. In other words, it lets you specify a font size in a relative fashion without being affected by the size of the parent, thereby eliminating compounding.

这个单位是相对html的font-size来说的，好了，实践的解决方法：根据实际的屏幕尺寸，动态计算顶级的font-size

而且最好做到换算起来简单，公式如下：

if (device-width > ui-width) {
    font-size: 100px;
} else {
    font-size: device-width * 100 / ui-width
}

设计稿: 640px(iphone 6p)
按钮: 120px;
屏幕尺寸: 500px;

计算：
font-size: 500 * 100 / 640;
button-width: 1.2rem( = 1.2 * 500 * 100 / 640 = 93.75px)

字体

字体不同于布局，自己不能随着尺寸变化而变化，或者说，只能大，不能小，因为太小了根本看不着，淘宝的字体大小是固定的。

Ratio 问题

还有一个问题，在不同的ratio下，1px的感觉是不一样的，比如，在一个ratio = 2 的分辨率下，会用两个像素点来作为1px
对图片的影响比较大，如果在一个100x100px的地方，放一个100x100px的图，在ration=1的情况下没问题，但是在ratio=2的情况下，就会显示模糊了，当然了，这个与标尺无关，需要提供额外的资源

window.devicePixelRatio; // mac 显示 为 2

CODE

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="initial-scale=1, maximum-scale=1, minimum-scale=1, user-scalable=no">
    <title>REM</title>
    <style>
        .btn-320 {
            width: 3.2rem;
            font-size: 14px;
        }
    </style>
    <script>
        var html = document.getElementsByTagName('html')[0];
        var width = html.clientWidth;
        var base = width > 1200 ? (1200 * 100 / 640) : (width * 100 / 640);
        html.style.fontSize = base + 'px';
    </script>
</head>
<body>
<div class="btn-320">button 320px</div>
</body>
</html>
